# 两类节点角色：算力节点 vs 安全共识节点

在 PulsWeb2 里，提供资源的参与者并不等同于“发币收益”叙事。这里更准确的说法是两类不同的**节点角色**：

## 1) 算力节点（Compute Node）

这类节点提供的是 **机密计算算力**，负责把用户的机密服务/机密任务真正跑起来。

- **你需要部署什么**：`tee-worker`（Worker 节点）
- **你提供什么**：CPU / 内存 /（可选）GPU，以及具备 TEE 能力的硬件环境（如 Intel SGX）
- **你的职责**：
  - 运行用户的机密容器/任务
  - 提供可验证的运行环境证明（Attestation）
  - 保持节点稳定在线

对应文档：
- 安装 Worker：见“安装 PulsWeb2 Worker”
- 证明与 DCAP：见“配置机密计算证书 (DCAP)”

## 2) 安全共识节点（Consensus Security Node）

这类节点提供的是 **安全与共识能力**，负责运行可信子网（Trusted Subnet），在其中完成关键的安全协作流程。

在当前架构中，`tee-dsecret` 是 **可信子网**：
- 它可以作为 **多个区块链主网的子网**（为不同主网提供同一套“可信子网能力”）。
- 它承载 DKG（分布式密钥生成）、密钥轮换、成员管理等需要共识保障的敏感流程。

- **你需要部署什么**：`tee-dsecret`（可信子网节点）
- **你提供什么**：可靠的网络与存储、稳定的运行环境，以及参与共识的节点能力
- **你的职责**：
  - 参与子网共识，保证协议一致性与抗作恶
  - 维护 DKG/Epoch 等关键流程的安全性与可用性
  - 为上层主网提供“可信子网服务”

对应文档：
- 部署 `tee-dsecret`：见“部署可信子网 `tee-dsecret`（安全共识节点）”

---

如果你只想提供算力、跑任务，部署 `tee-worker` 即可；  
如果你要参与安全共识、维护可信子网能力，需要部署 `tee-dsecret`。
